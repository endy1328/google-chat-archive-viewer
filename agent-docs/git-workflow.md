# Git 워크플로 초안

이 문서는 `google-chat-archive-viewer` 초기 단계에서 사용할 git 브랜치 전략과 운영 원칙을 정리한다.

## 목표

- 작은 팀에서도 충돌 없이 빠르게 작업한다.
- 배포 기준 브랜치를 단순하게 유지한다.
- 이슈 번호와 브랜치, PR, 커밋의 연결을 명확히 한다.

## 기본 브랜치 전략

- 기본 브랜치는 `main` 하나만 장기 유지한다.
- 기능 개발, 문서 작업, 버그 수정은 모두 짧게 유지되는 작업 브랜치에서 진행한다.
- 초기 단계에서는 `develop` 같은 장기 통합 브랜치를 만들지 않는다.

이 전략을 택한 이유는 현재 프로젝트 규모가 작고, 운영 문서 중심의 초기 세팅 단계이기 때문이다. 장기 브랜치를 늘리면 병합 비용만 커지고 기준선이 흐려진다.

## 브랜치 이름 규칙

- 기능: `feature/SUPAAA-32-google-chat-archive-viewer`
- 버그 수정: `fix/SUPAAA-32-short-description`
- 문서 작업: `docs/SUPAAA-32-short-description`
- 운영/설정: `chore/SUPAAA-32-short-description`

규칙:

- 접두사로 작업 유형을 먼저 드러낸다.
- 가능하면 Paperclip 이슈 번호를 포함한다.
- 설명은 짧고 소문자 kebab-case로 유지한다.

## 병합 규칙

- `main`에는 직접 커밋하지 않는다. 예외는 저장소 부트스트랩처럼 PR 기반 작업이 불가능한 초기 1회 정리뿐이다.
- 기본 병합 방식은 `Squash and merge`를 사용한다.
- Squash 커밋 제목은 `SUPAAA-32: 작업 요약` 형식으로 맞춘다.
- 작업 브랜치는 병합 후 삭제한다.

## 커밋 규칙

- 한 커밋은 하나의 의도만 담는다.
- 커밋 메시지 첫 줄에 이슈 번호를 포함한다.
- 예시: `SUPAAA-32: add initial git workflow docs`

## 보호 규칙

원격 저장소에서 아래 보호 규칙을 권장한다.

- `main` force-push 금지
- PR merge 전 최신 브랜치 기준 동기화 권장
- 최소 1회 리뷰 또는 셀프 체크리스트 확인 후 병합
- CI가 생기면 필수 상태 체크로 연결

## 작업 흐름

1. `main` 최신화
2. 이슈별 작업 브랜치 생성
3. 변경 작업 및 최소 검증
4. PR 생성
5. 리뷰 또는 셀프 체크 후 squash merge
6. 브랜치 삭제

## 확장 시점

아래 조건이 생기면 전략 재검토를 권장한다.

- 동시에 여러 기능 브랜치가 장기간 유지될 때
- 정기 배포 브랜치가 필요할 때
- 운영 환경과 스테이징 환경의 릴리즈 주기가 분리될 때

그 전까지는 `main` + 짧은 작업 브랜치 전략을 유지하는 편이 관리 비용이 가장 낮다.
